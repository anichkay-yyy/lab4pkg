# Дополнительная спецификация

## Нефункциональные требования

### Надёжность
- Все операции с БД используют prepared statements для защиты от SQL-инъекций
- Операция консолидации выполняется в транзакции (BEGIN/COMMIT/ROLLBACK) для обеспечения атомарности
- Ограничения целостности (CHECK, FOREIGN KEY, NOT NULL) обеспечивают корректность данных на уровне СУБД

### Удобство использования
- Меню-ориентированный интерфейс с нумерованными пунктами
- Разделение доступа по ролям: администратор видит всё, маклер — только свои данные
- Русскоязычный интерфейс с понятными сообщениями об ошибках
- Табличный вывод данных с заголовками колонок

### Производительность
- SQLite — встраиваемая СУБД, не требует отдельного сервера
- Использование индексов через PRIMARY KEY и FOREIGN KEY
- Prepared statements компилируются один раз и выполняются эффективно

### Сопровождаемость
- Модульная структура: каждый модуль отвечает за свою сущность (broker.c, goods.c, deal.c и т.д.)
- Заголовочные файлы с чистыми интерфейсами в папке includes/
- Unit-тесты для каждого модуля (30 тестов, CUnit)
- CI/CD через GitHub Actions — автоматическая сборка и тестирование

### Переносимость
- Код написан на стандартном C (C99)
- Зависимости: SQLite3, CUnit — доступны на Linux и macOS
- CI настроен для Ubuntu (GitHub Actions), локальная разработка на macOS

## Общие функциональные требования

- Хэширование паролей (DJB2) для безопасного хранения учётных данных
- Логирование ошибок SQL через stderr
- Поддержка работы с in-memory базой данных для тестирования
